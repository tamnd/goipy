# v0.1.3 — 2026-04-29

Subclass arithmetic for builtin types: `class N(int): pass; N(5) * 2`
returns 10, `N(5) + 1` returns 6, `S("hi") + " there"` returns `"hi there"`,
`isinstance(N(5), int)` is True, and `{N(5): "a"}[5]` returns `"a"`.

This is part 2 of three staged releases that close roadmap entry V2.
v0.1.4 lands the `typing.get_type_hints` fallout. Spec at
`notes/Spec/1500/1544_goipy_v0103_builtin_subclass_arith.md`.

## Why this matters

Before v0.1.3, `class N(int)` silently dropped the `int` base in
`__build_class__`, leaving N with no superclass and no inherited
behaviour:

```python
class N(int):
    def doubled(self):
        return self * 2

N(5) + 1                # goipy: TypeError       CPython: 6
N(5).doubled()          # goipy: TypeError       CPython: 10
isinstance(N(5), int)   # goipy: False           CPython: True
issubclass(N, int)      # goipy: False           CPython: True
```

Anything user code does with stdlib-shaped subclasses (counters that
inherit dict, NamedTuple-adjacent patterns, `Enum`-style int
subclasses, custom string types) failed at the first arithmetic op.

## Approach

Builtin types remain `*object.BuiltinFunc` — refactoring them to full
`*object.Class` would touch every dispatch site. Instead the subclass
carries a `BuiltinValue` payload built at construction time, and
operations that don't find a user-defined dunder unbox to the payload.

- `object.Class` gets `BuiltinBase string` + `BuiltinMRO []Object`.
- `object.Instance` gets `BuiltinValue Object`. For `class N(int)`,
  `N(5).BuiltinValue` is `*Int(5)`.
- `vm/ops.go` adds `unboxBuiltin(o)`: returns `inst.BuiltinValue` for
  Instances with a payload, else `o` unchanged.

`__build_class__` detects builtin-type BuiltinFunc bases, records the
base name on the new class, and synthesises a real MRO ending in
`object`. `call.go`'s class-call path constructs the BuiltinValue from
the args when no user `__init__` is defined.

After every dunder dispatch, the binary/unary/compare/contains/getitem
paths call `unboxBuiltin` on both operands. The type-specific Go
switches downstream then handle the unboxed values, producing plain
builtin results — which matches CPython: `N(5) + 1` returns plain
`int(6)`, not `N(6)`, unless `N` overrides `__add__`.

## What's in

- `object/object.go`: new fields on `Class` (`BuiltinBase`,
  `BuiltinMRO`) and `Instance` (`BuiltinValue`).
- `object/dict.go`: `Get` falls through from the `Str` fast path to
  `oHash` so a stored builtin-subclass `str` Instance still matches a
  plain `Str` key.
- `vm/ops.go`: `unboxBuiltin` helper. Numeric coercions (`asComplex`,
  `asIntOrFloat`, `toInt64`, `toBigInt`, `isComplex`) unbox at entry.
  `getAttr` on Instance falls back to the BuiltinValue when class MRO
  lookup misses, so `S("hi").upper()` works. `__mro__` and `__bases__`
  return `BuiltinMRO` slices when present. `unaryNeg` and `length`
  unbox.
- `vm/binops.go`: `add`, `sub`, `mul`, `truediv`, `floordiv`, `mod`,
  `pow`, `bitop`, `getitem` unbox after dunder dispatch.
- `vm/compare.go`: `compare`, `lt`, `contains` unbox.
- `vm/dunder.go`: `instReprHook`, `instStrHook`, `instTruthyHook`,
  `instHashHook`, `instEqHook` fall back to BuiltinValue when no user
  dunder is defined. So `repr(N(5))` is `5`, `hash(N(5)) == hash(5)`,
  `N(5) == 5` is True, and `bool(L([])) is False`.
- `vm/builtins.go`:
  - `__build_class__` detects builtin-type BuiltinFunc bases, records
    `BuiltinBase` and `BuiltinMRO`.
  - `int()`, `float()`, `abs()` unbox arguments after the dunder
    check.
  - `isinstance(N(5), int)` and `isinstance(N(5), object)` recognise
    builtin-base instances.
  - `issubclass(N, int)`, `issubclass(N, object)`, tuple-of-types
    form all walk the BuiltinBase relation.
- `vm/call.go`: class-call path auto-constructs `BuiltinValue` from
  the args when the class has a `BuiltinBase` and there is no user
  `__init__`. `intrinsicUnaryPos` unboxes.

## Out of scope (deferred)

- `typing.get_type_hints(f)` returning canonical types — V2 part 3 /
  v0.1.4.
- `repr(int) == "<class 'int'>"` — cosmetic; v0.1.4.
- `__init_subclass__`, slots, descriptor `__set_name__` for
  builtin-subclass instances — orthogonal to the surface this release
  fixes; opened only if a fixture demands it.
- Operator results that *preserve* the subclass type (`N(5) + N(1)`
  returning `N(6)` rather than `int(6)`) — CPython doesn't preserve
  unless the class overrides `__add__`. We match that.
