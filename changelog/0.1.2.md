# v0.1.2 — 2026-04-29

Builtin types behave as classes for `type()` / `isinstance()` /
`issubclass()` and expose `__mro__`, `__name__`, `__bases__`,
`__qualname__`, `__module__`.

This is part 1 of three staged releases that close roadmap entry V2.
v0.1.3 lands subclass arithmetic (`class N(int): ...; N(5) * 2`);
v0.1.4 lands the `typing.get_type_hints` fallout. Spec at
`notes/Spec/1500/1543_goipy_v0102_builtin_types_surface.md`.

## Why this matters

Before v0.1.2, `int`, `str`, `bool`, … were stored as
`*object.BuiltinFunc` and behaved like *functions*, not classes:

```python
type(int)             # goipy: <class 'function'>     CPython: <class 'type'>
isinstance(int, type) # goipy: False                  CPython: True
issubclass(bool, int) # goipy: False                  CPython: True
int.__mro__           # goipy: AttributeError         CPython: (int, object)
```

Frameworks that introspect type objects (pydantic, fastapi,
sqlalchemy, anything that compares `t is int` after a
`get_type_hints()` round-trip) saw the wrong shapes. Test code that
asserted `int.__mro__` raised AttributeError. Reflection-driven
container code that walks `cls.__bases__` failed.

## What's in

- `vm/ops.go`: `builtinTypeNames` set + `builtinTypeBases` map +
  `isBuiltinTypeName` / `isBuiltinSubclass` helpers. The MRO map
  has one entry today (`bool → (int,)`); the rest default to
  object.
- `vm/builtins.go`:
  - `wireBuiltinTypeAttrs` decorates each builtin-type BuiltinFunc
    with `__qualname__`, `__module__`, `__bases__`, `__mro__`.
    `__name__` is already returned by the existing short-circuit in
    `getAttr`.
  - `type` builtin is now self-referential: `type(int) is type`
    (same `*object.BuiltinFunc` returned). `type(C)` returns the
    same object for any user `*object.Class`.
  - `type(1)` returns the `int` BuiltinFunc rather than synthesizing
    a fresh bare class, so `type(1) is int` holds.
  - `isinstance(o, type)` recognises any builtin-type BuiltinFunc or
    `*object.Class` as a type.
  - `isinstance(int, object)` returns True.
  - `issubclass` walks the MRO map for builtin-type pairs;
    `issubclass(bool, int)`, `issubclass(int, object)`,
    tuple-of-types form all behave like CPython.

## Out of scope (deferred)

- `class N(int): pass; N(5) * 2` arithmetic — V2 part 2 / v0.1.3.
  The dispatch core still special-cases `*object.BuiltinFunc` for
  type identity; the subclass instance carries a Class, not the
  BuiltinFunc.
- `typing.get_type_hints(f)` returning canonical types — V2 part 3
  / v0.1.4. Annotation resolution doesn't yet round-trip the
  builtin-type identity.
- `repr(int) == "<class 'int'>"` — cosmetic; current repr is
  `<built-in function int>`. Slated for v0.1.4.
- `int.__class__` returning `type` — interferes with the
  BuiltinFunc.__class__ surface from v0.0.335. v0.1.4.
- `type("Name", (Base,), {})` 3-arg form — separate fixture if
  required; today raises a clean `TypeError`.

## Fixture

`internal/testdata/348_builtin_types_surface.py` — seven sections:
`type(x) is T`, `type(T) is type`, `isinstance(T, type)`,
`issubclass` matrix, `__name__` / `__qualname__` / `__module__`,
`__mro__` for every builtin type, `__bases__` for the same set.
Output is pure ascii booleans and small tuples — no wall-clock,
no hash-dependent ordering.
