# v0.1.1 — 2026-04-29

threading: `Condition.wait(timeout)` / `Event.wait(timeout)` /
`Semaphore.acquire(timeout)` / `Barrier.wait(timeout)` no longer
deadlock the goroutine scheduler.

First release of the v0.1.x cycle. Closes entry V1 from the
v0.1.x roadmap at `notes/Spec/1500/1541_goipy_v01x_roadmap.md`.
Spec at `notes/Spec/1500/1542_goipy_v0101_condition_timeout.md`.

## Why this matters

Before v0.1.1, every `threading.Condition.wait(timeout=…)`,
`Event.wait(timeout=…)`, and `Semaphore.acquire(timeout=…)` call
hung on `sync.Cond.Wait`, which has no timeout. Go's runtime
deadlock detector then killed the process with "all goroutines
are asleep — deadlock!". Producer/consumer code, asyncio bridges,
back-pressure loops, and "wait at most N seconds" patterns all
crashed silently in tests and lethally in prod.

## What's in

- `vm/stdlib_threading.go`: new `chCond` helper — a channel-backed
  condition variable that supports timed waits. Each `Wait()`
  registers a fresh channel; FIFO of waiters; `Notify(n)` wakes
  up to n in order; `Broadcast()` wakes all; timeout via `select`
  + `time.After`.
- `parseTimeout(o)` helper: maps a Python `timeout` argument
  (None / float seconds / negative / missing) to a Go
  `time.Duration`. None or negative → -1 (no timeout).
- `Event.wait(timeout=None)` honours timeout; returns the flag's
  current value on timeout.
- `Condition.wait(timeout=None)` returns `True` if notified,
  `False` on timeout (matches CPython 3.11+).
- `Condition.wait_for(predicate, timeout=None)` re-evaluates
  predicate on each wake; on overall timeout, returns the
  predicate's last result (CPython semantics).
- `Condition.notify(n)` honours `n` (was always Signal-one before;
  now wakes up to n waiters).
- `Semaphore.acquire(blocking, timeout)` honours `timeout` kwarg
  and positional second arg.
- `Barrier.wait(timeout=None)` honours `timeout`. On timeout the
  barrier is broken (matches CPython); other parties wake and see
  `BrokenBarrierError`.
- New: `threading.BrokenBarrierError`, a subclass of
  `RuntimeError`. `Barrier.wait` raises this instead of a bare
  `RuntimeError`.

## Out of scope

- `RLock` over-release should raise `RuntimeError` but currently
  doesn't — that's V9 in the roadmap, scheduled for v0.1.8.
- `Lock` / `RLock` `acquire(timeout)` — already supported via
  the existing `blocking` arg + Go scheduler; not part of this
  cycle.
- `queue.Queue.get(timeout)` / `put(timeout)` — separate module,
  separate spec if broken (probe didn't surface a deadlock there).
- asyncio Lock/Event/Condition timeouts — those run on the
  asyncio event loop, separate codepath.
