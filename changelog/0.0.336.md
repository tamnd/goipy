# v0.0.336 — 2026-04-29

Hash invariant

The v0.0.334 deep gap audit
(`notes/Spec/1500/1530_goipy_v01_gap_audit.md`) flagged B8: goipy's
numeric hashes broke Python's data-model promise that `x == y` implies
`hash(x) == hash(y)`. `hash(1)` returned `-7046029254386353132`,
`hash(1.0)` returned `1000000` — equal values, different hashes. A
dict keyed by `1` could not be looked up by `1.0`. Spec at
`notes/Spec/1500/1532_goipy_v0036_hash_invariant.md`.

- `object/dict.go` gained three CPython-faithful helpers:
  `HashFloat64` (the `_Py_HashDouble` frexp/2^28 algorithm),
  `HashBigInt` (`|x| mod (2^61 - 1)` with sign), and `HashRational`
  (modular inverse over the same Mersenne prime, for any
  `num/den` pair). All three return `int64` in CPython's exact
  numeric range.
- The existing `Hash()` switch routes `*Int`, `*Float`, `*Complex`
  through the new helpers. `hash(1) == hash(1.0) == hash(1+0j) == 1`.
- `Fraction.__hash__` (`vm/stdlib_fractions.go`) and
  `Decimal.__hash__` (`vm/stdlib_decimal.go`) now feed their
  numerator/denominator into `HashRational`. `hash(Fraction(1,1))`,
  `hash(Decimal(1))`, `hash(Decimal('1.0'))` all return `1`. The
  previous Decimal hash returned `Float64bits(value)` — a unique-by-
  representation but invariant-violating value.
- `Decimal` infinities hash to ±314159 (matching float); signaling NaN
  raises TypeError per CPython 3.10+. Quiet NaN hashes to 0 (CPython
  randomises per-instance; goipy keeps stable for now).
- New fixture `internal/testdata/336_hash_invariant.py` — covers the
  trio invariant, cross-type dict lookup, set dedup, big-int
  boundaries (`2**61-1`, `2**61`, `2**100`), `±inf`, `nan`,
  `Decimal('1.0')`, `Fraction(2, 4)`, and `bool ↔ int` parity.

Out of scope: hash randomisation (PYTHONHASHSEED). CPython randomises
string hashes by default; goipy's string hash stays deterministic
(FNV-1a). Per-instance NaN hash randomisation also stays at 0 — code
rarely depends on distinguishing NaN hashes.
