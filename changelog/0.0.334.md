# v0.0.334 — 2026-04-28

`subprocess` shell handling

The v0.0.331 v0.1.0-prep audit listed `subprocess shells through sh -c`
as out of scope. This release closes that gap. Spec at
`notes/Spec/1400/1479_goipy_v0034_subprocess_shell.md`.

- `Popen(..., shell=True)` with a list/tuple `args` now follows
  CPython: `argv[0]` is the script string, `argv[1:]` are passed as
  positional arguments to the shell, surfacing as `$0`, `$1`, … inside
  the script. Previously goipy joined every entry with a space and
  fed the result back to `sh -c`, collapsing positional structure.
- New `executable=` kwarg. In `shell=True` mode it picks the shell
  binary (defaulting to `/bin/sh` on Posix, `COMSPEC`/`cmd.exe` on
  Windows). In `shell=False` mode it sets `cmd.Path` directly while
  preserving `argv[0]` as supplied by the caller — matching CPython's
  argv0-rewrite trick.
- Shell selection now picks the platform shell. Posix stays on
  `/bin/sh`; Windows uses `COMSPEC` (or `cmd.exe`) with `/c`.
  `getoutput` and `getstatusoutput` route through the same selector.
- New fixture `internal/testdata/334_subprocess_shell.py` — covers
  shell=True with list args, shell=True classic str, shell=True with
  pipes, `executable=` in both modes, and `getoutput` /
  `getstatusoutput`. Verified against CPython 3.14.

Out of scope: single-string `Popen('echo hi')` strict CPython
semantics (we still split on whitespace, friendlier than CPython's
"the whole string is the program path"); `posix=True/False` toggling
`shlex` parsing; `start_new_session`, `preexec_fn`, `pass_fds`. Each
of those wants its own release.
