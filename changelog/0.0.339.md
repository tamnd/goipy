# v0.0.339 — 2026-04-29

Async generators

The v0.1.0 gap audit (`notes/Spec/1500/1530_goipy_v01_gap_audit.md`)
flagged B4: `async def foo(): yield x` produced an
`*object.Generator`, but the type didn't expose `__aiter__` /
`__anext__`. `async for v in foo()` failed at `GET_AITER` with
`TypeError 'generator' object is not an async iterable`. The
bytecode side has been ready since v0.0.284
(`INTRINSIC_ASYNC_GEN_WRAP`); the missing piece was the runtime
async-iterator protocol. Spec at
`notes/Spec/1500/1533_goipy_v0039_async_generators.md`.

- `vm/generator.go` `genMethod` now resolves `__aiter__`,
  `__anext__`, `asend`, `athrow`, and `aclose` for generators
  whose code carries `CO_ASYNC_GENERATOR` (`Code.Flags & 0x0200`).
  Detection is via the new `isAsyncGen(gen)` helper.
- `__anext__` (and `asend`/`athrow`) return a one-shot awaitable
  built by `makeAsyncGenStep`. The awaitable is an `*object.Iter`
  whose `Next()` calls `resumeGenerator` (or `throwGenerator`) once
  and reshapes the outcome:
  - gen yielded `v` → raise `StopIteration(v)` so `await` lifts the
    value to the caller frame.
  - gen returned → raise `StopAsyncIteration` so `async for`
    terminates cleanly.
  - other exception → propagate.
- `aclose` uses `makeAsyncGenClose`, a separate awaitable that
  throws `GeneratorExit` and treats `StopIteration` /
  `GeneratorExit` / `StopAsyncIteration` as a successful close
  (returns `None` to the caller). Other exceptions propagate.
- New fixture `internal/testdata/339_async_generators.py` —
  `async for`, `asend` driving a state machine, `aclose`,
  `athrow` injection, async comprehension, and two interleaving
  agens.

Out of scope: a separate `*object.AsyncGenerator` type (current
Generator with the CO_ASYNC_GENERATOR flag is enough), `agen.ag_*`
introspection (gap F2, deferred to v0.0.343), and
`sys.set_asyncgen_hooks` integration.
