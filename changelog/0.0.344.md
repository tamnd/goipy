# v0.0.344 — 2026-04-29

`iter(callable, sentinel)` + `slice.indices(n)` + `functools._CacheInfo`

Closes groups **F4**, **F5**, and **C5** from the v0.1.0 audit. Three
small surface fixes in one release. Spec at
`notes/Spec/1500/1538_goipy_v0044_iter_slice_cacheinfo.md`.

- `vm/builtins.go` `iter` builtin: two-arg form returns an iterator
  that calls `a[0]()` until the result equals `a[1]`.
- `vm/ops.go` `*object.Slice.indices(n)`: returns the
  `(start, stop, step)` tuple normalised against `n`. Mirrors
  CPython's `PySlice_GetIndices` — clamps to `length`/`length-1` for
  positive/negative step, fills `None` with the right defaults, and
  raises `ValueError` on `step == 0`.
- `vm/stdlib_functools.go` `lru_cache` `cache_info()` now returns a
  `_CacheInfo(hits, misses, maxsize, currsize)` namedtuple instance
  built once via the shared `makeNamedTuple` helper. Supports both
  attribute access (`info.hits`) and tuple-style indexing
  (`info[0]`). `_CacheInfo` itself is exposed on the functools module
  dict so libraries can `isinstance` against it.
- New fixture `internal/testdata/344_iter_slice_cacheinfo.py` —
  iter sentinel reaches/never-reaches/empty cases, slice.indices on
  positive/negative/None bounds and step==0 raise, lru_cache cache_info
  attribute + tuple access, equality against a plain tuple.

Out of scope: `slice.start/stop/step` already exposed (verified by
prior audits); `iter()` zero-arg form not part of the language.
